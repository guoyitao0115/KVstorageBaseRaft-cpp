## 1. 先把之前学过的关键内容唤醒

你之前的学习主线非常清晰：先围绕 Raft 选举和日志复制，把共识层的核心链路走了一遍，然后进入持久化、快照和 KvServer 状态机层。现在重新开始时，不需要把旧日志全文重读，先恢复下面这些“骨架模型”就够了。

### 1.1 项目分层模型

一台 server 进程里同时包含几层：

```text
Clerk 客户端
  -> KvServer：接收 Get / Put / Append RPC
  -> Raft：复制日志、选主、commit
  -> applyChan：Raft 把已 commit 的日志推给 KvServer
  -> SkipList：真正执行命令的 KV 状态机
  -> Persister：保存 raft state 和 snapshot
```

关键边界：

- `KvServer` 面向客户端，负责线性一致性等待、去重、执行状态机、做快照。
- `Raft` 不理解业务含义，只复制 `Op` 序列化后的命令字符串。
- `SkipList` 是最终状态机；只要所有节点按同一顺序 apply 相同命令，状态就一致。
- `Persister` 是底层文件读写器；Raft 和 KvServer 都会间接依赖它。

### 1.2 Raft 的三条后台链路

`Raft::init` 启动三条长期循环：

```text
leaderHearBeatTicker  -> doHeartBeat       -> sendAppendEntries / leaderSendSnapShot
electionTimeOutTicker -> doElection        -> sendRequestVote
applierTicker         -> getApplyLogs      -> applyChan.Push
```

前两个跑在 `monsoon::IOManager` 协程调度器里，`applierTicker` 用独立线程。原因是心跳和选举 ticker 主要 sleep，适合协程；apply 可能被 KvServer 的状态机和快照拖慢，用线程更稳。

### 1.3 选举链路

需要重新记住的函数闭环：

```text
electionTimeOutTicker
  -> doElection
    -> sendRequestVote
      -> 对端 RequestVote
        -> UpToDate
```

关键点：

- `m_lastResetElectionTime` 是“选举看门狗”的喂狗时间。
- 喂狗时机包括：收到合法 AE、给别人投票、自己刚发起选举。
- `doElection` 做四件事：转 Candidate、term++、自投、并发发 RequestVote。
- `RequestVote` 的安全核心是 `UpToDate(lastLogIndex, lastLogTerm)`。
- `sendRequestVote` 收到更高 term 必须三变：`status=Follower`、更新 `currentTerm`、`votedFor=-1`，再持久化。

### 1.4 日志复制链路

需要重新记住的函数闭环：

```text
KvServer::Get / PutAppend
  -> Raft::Start
  -> doHeartBeat
  -> sendAppendEntries
  -> 对端 AppendEntries1
  -> leader 推进 commitIndex
  -> applierTicker
  -> KvServer::ReadRaftApplyCommandLoop
```

关键点：

- 心跳和日志复制都是 `AppendEntries`，空 `entries` 就是心跳。
- Leader 给每个 Follower 发的 AE 不同，因为每个 peer 的 `nextIndex[i]` 不同。
- `prevLogIndex / prevLogTerm` 是日志匹配锚点。
- Follower 不能粗暴“截断再追加”，因为旧 AE 重放可能删掉已经存在的新日志；本项目使用逐条幂等处理。
- `matchIndex` 用来判断复制进度，`nextIndex` 用来决定下次从哪里发。
- `sendAppendEntries` 推进 commit 时必须注意 Figure 8 规则：只有当前 term 的日志被多数派复制后，Leader 才能直接推进 commit。

### 1.5 commit 和 apply 不要混

```text
写入本地日志：m_logs.push_back + persist
复制成功：多数派 AppendEntries 成功
commit：Leader 推进 m_commitIndex
apply：applierTicker 推 ApplyMsg，KvServer 执行到 SkipList
```

不变量：

```text
lastApplied <= commitIndex <= lastLogIndex
```

客户端真正收到 `OK`，发生在 KvServer 等到对应 `raftIndex` 的命令被 apply 后。

### 1.6 快照边界模型

快照后，`m_logs` 不再从全局 index 1 开始。必须始终区分：

- 逻辑 index：Raft 全局日志编号，例如 101。
- 物理下标：`m_logs` vector 里的下标，例如 0。

转换核心：

```cpp
sliceIndex = logIndex - m_lastSnapshotIncludeIndex - 1;
```

相关函数：

- `getSlicesIndexFromLogIndex`
- `getLogTermFromLogIndex`
- `getLastLogIndexAndTerm`
- `Snapshot`
- `InstallSnapshot`
- `leaderSendSnapShot`

### 1.7 KvServer 的线性一致性等待模型

`KvServer::Get` 和 `KvServer::PutAppend` 都会：

```text
构造 Op
  -> Raft::Start(op) 返回 raftIndex
  -> waitApplyCh[raftIndex] 等待 apply
  -> ReadRaftApplyCommandLoop 收到 ApplyMsg
  -> GetCommandFromRaft / SendMessageToWaitChan 唤醒 RPC 线程
```

必须检查“这个 index 最终 commit 的是不是我这条命令”，因为旧 Leader 追加的日志可能被新 Leader 覆盖。

`ClientId + RequestId` 是去重表的核心。Put/Append 不能重复执行，Get 可重复读但仍会记录请求序号。

## 4. 建议的“读完全部代码”打卡表

1. 复述一次 Put 请求全链路：`Clerk -> KvServer -> Raft::Start -> AE -> commit -> apply -> SkipList -> reply`。
2. 复述一次 Leader 选举链路：ticker、term、自投、RequestVote、UpToDate、多数派。
3. 手画 `nextIndex/matchIndex` 在一次失败 AE 后如何变化。
4. 单独解释 `getSlicesIndexFromLogIndex`，用一个快照到 index=100 的例子说明。
5. 读完 KvServer，说明 `waitApplyCh` 和 `m_lastRequestId` 各解决什么问题。
6. 读完 RPC，说明一个 protobuf Stub 调用为什么最后会进入 `RpcProvider::OnMessage`。
7. 读完 SkipList，自己修复并验证 `load_file` 的 value 恢复问题。
8. 读完 Persister，设计一个真实崩溃恢复实验，验证构造函数清空文件的问题。
9. 读完 fiber，解释 `usleep` 在 ticker 中为什么能让出协程而不是卡死线程。
10. 最后做一次代码审查：列出 5 个潜在 bug 或可优化点，并给出最小修复方案。

## 5. 当前值得重点验证的疑点

这些不是必须立刻修，但适合在后续学习中作为练习题：

| 疑点 | 位置 | 为什么值得看 |
| --- | --- | --- |
| `SkipList::load_file` 疑似把 value 恢复成 key | `src/skipList/include/skipList.h` | 会影响 snapshot 恢复正确性 |
| `ExecuteAppendOpOnKVDB` 当前像 Put，不像 Append | `src/raftCore/kvServer.cpp` | Append 语义可能不完整 |
| `Persister::Persister` 构造时清空文件 | `src/raftCore/Persister.cpp` | 与“崩溃后 readPersist 恢复”目标冲突 |
| `CondInstallSnapshot` 直接返回 true | `src/raftCore/raft.cpp` | 快照安装缺少有效性判断 |
| `IfNeedToSendSnapShotCommand` 没用 `proportion` 参数 | `src/raftCore/kvServer.cpp` | 快照阈值逻辑可读性差 |
| `votedNum / appendNums` 是 `shared_ptr<int>` | `src/raftCore/raft.cpp` | 当前靠大锁兜底，语义上更适合 atomic 或明确锁保护 |
| `leaderUpdateCommitIndex` 未作为主线使用 | `src/raftCore/raft.cpp` | 可比较两种 commit 推进实现 |
| `KvServer::StartKVServer` 只有声明 | `src/raftCore/include/kvServer.h` | 可清理或补实现 |
| `Semaphore`、`Fiber::TotalFiberNum`、`Fiber::GetCurFiberID` 不完整 | `src/fiber` | 协程库可完善点 |
