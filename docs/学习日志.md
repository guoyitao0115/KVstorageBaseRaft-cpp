## Raft::persistData

### 函数位置

- 文件：`src/raftCore/raft.cpp`
- 起始行：1049
- 函数签名：

```cpp
std::string Raft::persistData()
```

### 函数作用

`Raft::persistData()` 的作用是把当前 Raft 节点需要持久化的核心状态序列化成一个 `std::string`，供 `persist()` 或快照相关逻辑交给 `Persister` 写入本地文件。

它不是直接写文件，而是只负责“打包数据”。真正写入磁盘的是 `Persister`。

整体流程可以理解为：

```text
Raft 内存状态
  -> persistData() 序列化成字符串
  -> persist() / m_persister->Save(...)
  -> Persister 写入本地持久化文件
```

### 代码讲解

```cpp
std::string Raft::persistData() {
  BoostPersistRaftNode boostPersistRaftNode;
```

这里创建了一个临时对象 `boostPersistRaftNode`。它是 `Raft` 类内部定义的辅助类，专门用来承载需要被 Boost.Serialization 序列化的字段。

```cpp
  boostPersistRaftNode.m_currentTerm = m_currentTerm;
  boostPersistRaftNode.m_votedFor = m_votedFor;
  boostPersistRaftNode.m_lastSnapshotIncludeIndex = m_lastSnapshotIncludeIndex;
  boostPersistRaftNode.m_lastSnapshotIncludeTerm = m_lastSnapshotIncludeTerm;
```

这里把 Raft 节点当前的关键状态复制到临时对象中：

- `m_currentTerm`：当前任期。
- `m_votedFor`：当前任期已经投票给哪个节点。
- `m_lastSnapshotIncludeIndex`：快照中包含的最后一条日志的 index。
- `m_lastSnapshotIncludeTerm`：快照中包含的最后一条日志的 term。

这些字段必须持久化。否则节点宕机重启后，可能忘记当前任期、重复投票，或者无法正确判断快照边界处的日志匹配关系。

```cpp
  for (auto& item : m_logs) {
    boostPersistRaftNode.m_logs.push_back(item.SerializeAsString());
  }
```

`m_logs` 中保存的是 Protobuf 类型的日志条目 `raftRpcProctoc::LogEntry`。这里没有直接让 Boost 序列化 Protobuf 对象，而是先调用 Protobuf 自己提供的：

```cpp
item.SerializeAsString()
```

把每条日志序列化成字符串，然后放入 `BoostPersistRaftNode::m_logs` 这个 `std::vector<std::string>` 中。这样 Boost 只需要处理普通字符串数组，逻辑更简单。

```cpp
  std::stringstream ss;
  boost::archive::text_oarchive oa(ss);
  oa << boostPersistRaftNode;
  return ss.str();
}
```

这几行使用 Boost.Serialization 做最终序列化：

- `std::stringstream ss`：创建一个字符串流，用来接收序列化结果。
- `boost::archive::text_oarchive oa(ss)`：创建文本格式的输出归档器，并绑定到 `ss`。
- `oa << boostPersistRaftNode`：把 `boostPersistRaftNode` 按照它的 `serialize()` 函数规则写入归档器。
- `return ss.str()`：返回序列化后的完整字符串。

其中 `oa << boostPersistRaftNode` 会调用 `BoostPersistRaftNode` 内部的序列化函数：

```cpp
template <class Archive>
void serialize(Archive &ar, const unsigned int version) {
  ar &m_currentTerm;
  ar &m_votedFor;
  ar &m_lastSnapshotIncludeIndex;
  ar &m_lastSnapshotIncludeTerm;
  ar &m_logs;
}
```

也就是说，Boost 会按这个顺序把字段写入字符串。

### 学习过程中遇到的问题和解答

问题 1：`boost::archive::text_oarchive oa(ss);` 是什么意思？

解答：这是创建一个“文本输出归档器”。它负责把 C++ 对象序列化成文本格式，并把结果写入 `ss` 这个字符串流。

问题 2：`oa << boostPersistRaftNode;` 是普通的输出操作吗？

解答：不是普通的 `cout << obj`。这里的 `<<` 是 Boost.Serialization 提供的序列化操作。执行时会查找并调用 `BoostPersistRaftNode` 的 `serialize()` 函数，把对象中的字段依次写入归档器。

问题 3：为什么 `m_logs` 要先 `SerializeAsString()`？

解答：因为 `m_logs` 的元素是 Protobuf 生成的 `LogEntry` 对象。项目选择让 Protobuf 负责序列化单条日志，再让 Boost 序列化 `vector<string>`。这样避免了直接让 Boost 处理复杂的 Protobuf 类型。

问题 4：为什么 Raft 必须持久化这些字段？

解答：Raft 安全性要求节点崩溃恢复后仍然记得当前任期、投票记录和日志。比如 `m_votedFor` 如果丢失，节点可能在同一个 term 给多个 Candidate 投票，从而破坏选举安全性。

### 可以提升或优化的地方

1. 可以考虑统一序列化方案。

当前实现是 “Protobuf 序列化日志条目 + Boost 序列化整体 Raft 状态”。这能工作，但同时引入两套序列化机制。后续可以考虑全部使用 Protobuf 定义持久化结构，降低心智负担。

2. 可以增加序列化失败处理。

当前函数默认序列化一定成功。如果 Boost 序列化抛异常，调用方没有明显处理。后续可以在持久化链路中增加异常捕获和日志记录。

3. 可以注意文本归档的体积和性能。

`text_oarchive` 可读性较好，但体积通常比二进制序列化更大。Raft 日志增长较快时，可以考虑 `binary_oarchive` 或纯 Protobuf 编码。

4. 可以把持久化字段说明集中写清楚。

这个函数依赖 `BoostPersistRaftNode::serialize()` 的字段列表。后续维护时，如果新增必须持久化的 Raft 状态，需要同时更新辅助类和恢复逻辑 `readPersist()`，否则容易出现恢复状态不完整的问题。
